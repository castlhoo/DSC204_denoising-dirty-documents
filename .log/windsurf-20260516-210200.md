### Identified Weaknesses (5 root causes)

**#1 — Premature early stopping (most critical):**
`patience=4` with `ReduceLROnPlateau` fired at epoch 2. At epoch 1, val RMSE = 0.148; at epoch 2, it was 0.097 — the scheduler saw this one-step improvement as "good enough" and began its patience countdown. The model essentially never trained. This single bug accounts for the majority of the performance gap.

**#2 — Model too small:**
`base_channels=16` → ~488K parameters. Document denoising requires recognizing stain gradients, stroke width, background color regions, and texture — simultaneously. 488K parameters cannot represent this complexity. The AI agent was under-parameterized for the task.

**#3 — No background normalization:**
`image / 255.0` treats all images identically. A coffee-stained page (background ~0.6) and a clean page (background ~0.95) both appear in [0,1] range, but the model needs to learn completely different corrections for each. Without background normalization, the same input value means different things in different images — the model cannot learn a consistent mapping.

**#4 — Residual output cap:**
`0.1 * tanh(x)` limits correction to ±0.1. For a pixel that should be 1.0 (white background) but is currently 0.5 (stained), the model can only push it to 0.6 — still far from correct. This was a misguided stability measure that severely restricted model capability.

**#5 — Low data utilization:**
115 images × 2 patches = 230 patches/epoch. With batch size 16, that is only ~14 gradient update steps per epoch. Over 25 epochs (but really only 2), the model received fewer than 30 weight updates total. This is insufficient for a 488K-parameter network to learn anything meaningful.

**Outcome:** All 5 weaknesses identified, each with a clear and implementable fix. The U-Net architecture itself is sound — only the configuration and preprocessing are broken.

### Architecture Decision: `ImprovedUNet`

Replaced 3-level ResidualUNet with a 4-level ImprovedUNet:

```
Input (1ch)
  → Enc1: Conv→BN→ReLU→Conv→BN→ReLU  [1→32ch,  64×64]
  → MaxPool → Enc2: Conv block          [32→64ch, 32×32]
  → MaxPool → Enc3: Conv block          [64→128ch,16×16]
  → MaxPool → Bottleneck: Conv block    [128→256ch,8×8]
  ↑ Upsample + Skip(128ch) → Dec3      [384→128ch,16×16]
  ↑ Upsample + Skip(64ch)  → Dec2      [192→64ch, 32×32]
  ↑ Upsample + Skip(32ch)  → Dec1      [96→32ch,  64×64]
  → Output Conv 1×1 → clamp(0,1)       [32→1ch,   64×64]
```

**Why 4 encoder levels?** Each MaxPool halves spatial dimensions. 4 levels gives the bottleneck a receptive field covering the entire 64×64 patch, enabling global context (background color, stain extent). 3 levels only reaches 8×8 bottleneck — insufficient for large stain regions.

**Why BatchNorm?** Small dataset (115 images) = high risk of unstable gradients. BatchNorm normalizes each layer's output to zero mean and unit variance, dramatically stabilizing training. This was absent in the AI agent's code.

**Why remove the 0.1 × tanh cap?** The cap limits corrections to ±0.1 per pixel — completely arbitrary and too conservative. Direct prediction with `clamp(0,1)` allows corrections of any magnitude. The model decides how much to correct, not the programmer.

**Why base_channels=32?** Double the channels → ~4× parameters (488K → 1.95M). This is the maximum that remains practically fast on CPU. Going to 64 would give 7.7M parameters but training would take 8+ hours.

**Total parameters: 1,949,121** — 4× the AI agent's model.

### Preprocessing Decision: Otsu Background Normalization

**The problem:** Dirty documents have wildly different background brightness. A coffee-stained page might have background at 0.6; a yellowed page at 0.75; a clean page at 0.95. Without correcting for this, the same pixel value means different things in different images. The model cannot learn a consistent correction rule.

**Why Otsu's method?**
Otsu (1979) is a classical algorithm that automatically finds the optimal threshold separating dark pixels (text) from bright pixels (background) by maximizing inter-class variance. For document images, this perfectly separates the two pixel populations.

**Implementation (from scratch, no scikit-image):**
1. Compute 256-bin histogram of pixel intensities
2. For each possible threshold t (0–255): compute inter-class variance between pixels below t (text) and pixels above t (background)
3. Select t* that maximizes inter-class variance
4. Compute mean brightness of all pixels above t* (= background estimate)
5. Divide entire image by that background estimate → background ≈ 1.0, text proportionally darker

**Result:** Every image, regardless of staining severity, enters the model with background ≈ 1.0. The model only needs to learn one mapping, not 115 different ones.

## Entry 7 — Iteration 2: TTA + CosineAnnealingLR + 80 Epochs

**Time:** 2026-05-16T21:50:00Z
**User prompt:** "I'd like to beat the Gold threshold if possible. Make the log more detailed, and everything in English."

**Context:** val RMSE = 0.01734 after 40 epochs. Gold threshold is 0.01649 — only 5% away. The key insight from Entry 6 was that LR never reduced (ReduceLROnPlateau never triggered), meaning the optimizer was still taking large steps at epoch 40. The model had not fully converged. Two targeted improvements were added.

**Improvement 1 — Test-Time Augmentation (TTA):**
At prediction time, run the model on 4 variants of each image and average the results:
- Original
- Horizontally flipped (then flip prediction back)
- Vertically flipped (then flip prediction back)
- Both flips combined

Averaging 4 predictions reduces random variance from the CNN's positional sensitivity. This costs 4× inference time but no additional training. Expected gain: 5–15% RMSE reduction.

**Bug encountered:** NumPy's `np.fliplr()` / `np.flipud()` return array views with negative strides. PyTorch cannot create tensors from negative-stride arrays. Fix: add `.copy()` after every flip operation to create a contiguous copy before converting to tensor.

**Improvement 2 — CosineAnnealingLR:**
Replaced `ReduceLROnPlateau` with `CosineAnnealingLR(T_max=80, eta_min=1e-6)`. 

Reasoning: `ReduceLROnPlateau` only reduces LR when val RMSE stops improving. In Entry 6, it never triggered because improvements kept coming — so LR stayed at 1e-3 for all 40 epochs. The optimizer was taking large steps the entire time, which is why epoch 40 overshot (0.023 vs epoch 39's 0.017). `CosineAnnealingLR` forces a smooth decay from 1e-3 → 1e-6 over the full 80 epochs regardless of val performance, enabling fine-grained convergence in later epochs when the model is near the optimum.

**80 epochs + patience=15:** Doubling epochs gives the model more steps at low LR (cosine tail). Patience=15 prevents early stopping from interfering with the longer schedule.

**Files modified:** `train_sean.py`
- `predict_single()`: added TTA 4-way averaging with `.copy()` fix
- Scheduler: `ReduceLROnPlateau` → `CosineAnnealingLR(T_max=80, eta_min=1e-6)`
- Default epochs: 40 → 80, default patience: 8 → 15
